# proto-schemaorg — TODO

Roadmap for turning the schema.org vocabulary into a grammar + protobuf type
system and extracting schema.org structured data into it. Tracked in the repo and
mirrored as a GitHub issue.

## Done

- [x] **schema.org vocab → committed proto type system.** `cmd/genast` builds a
  gluon grammar AST directly from the vendored JSON-LD (`schema/`) and lowers it
  with gluon's compiler → `proto/schemaorg.{proto,fdset}` (~1841 messages,
  `-all`). No EBNF-text / markup-terminal machinery. (ADR 0001.)
- [x] **Human-readable grammar view.** `lang/schemaorg.ebnf`, rendered by gluon's
  `metaparser.RenderEBNF` (LexDescriptor-driven; added upstream in
  accretional/gluon#10 / #11), with the subClassOf hierarchy annotated.
- [x] **JSON-LD extractor.** `service/jsonld.Extract` parses raw JSON-LD via
  **proto-json** (`jsonparse.Parse` → `json.proto` AST) and fills `Schema<Type>`
  messages by reflection. No `encoding/json`.

## Next

### Parsing (through the grammar repos — never byte-level here)

- [x] **proto-html parser library** (proto-html PR #2). Gluon-based HTML parser
  (`htmlparse.ParseDOM`) with a lexical token-matcher layer + inter-element
  whitespace tolerance, projecting to a generic DOM. Not `x/net/html`.
- [x] **HTML → JSON-LD plumbing** (`jsonld.ExtractHTML`). Parses a page via
  proto-html, walks the DOM for `<script type="application/ld+json">` blocks, and
  feeds each to `jsonld.Extract`. (Depended on proto-json #1 / proto-html #3, both
  fixed.)
- [x] **Microdata extractor** (`service/microdata`). The WHATWG DOM-walk
  algorithm (`docs/MICRODATA_SPEC.md`) over proto-html's DOM → the WHATWG
  structured form (items of {type, id, properties}; value = string | nested
  item), with itemref, cycle guards, the value cascade, multi-token
  itemprop/itemtype, and URL resolution. Gated against
  `testing/testdata/synthetic/` — **13/14 pass**; `013` is skipped pending
  proto-html#6 (bare `<meta itemprop>` parse gap), not an extraction bug.
- [x] **Unified item→message mapping.** `schemaorg.Item`/`Value` is the one
  syntax-independent shape both front-ends produce, and `schemaorg.Build` is the
  single mapper to `Schema<Type>` messages (property wrappers, oneof arms,
  datatype vs item fields). `jsonld` builds an `Item` from the JSON AST then
  `Build`s it; `microdata.Messages` does the same over its DOM extraction. No
  duplicated mapping logic.

### Type system / grammar

- [ ] **Enumerations as value constraints.** Enum-ranged properties currently
  lower to nested items; model them as an alternation of members instead.
- [ ] **Reference overlay.** Resolve the out-of-context cross-references a CF
  grammar can't — microdata `itemref`, JSON-LD `@id`/`@reverse` (a graph, not a
  tree), IDREF-style links — in a second pass (ADR 0001).
- [ ] **subClassOf structurally.** Inheritance is flattened into each type's
  fields and only *annotated* in the EBNF; consider a base-group production per
  supertype so the hierarchy is in the type system, not just a comment.
- [ ] **Digit-leading types.** `3DModel` etc. are dropped by `isIdent` in
  `genast` (invalid proto identifiers). Handle with a prefix/mangling scheme.
- [ ] **Custom `@context`.** JSON-LD extraction assumes the default schema.org
  context; handle vocab prefixes / remote contexts.
