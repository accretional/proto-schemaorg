# CLAUDE.md — proto-schemaorg

proto-schemaorg turns the **schema.org vocabulary** into a **grammar** and a
**protobuf type system**, and extracts schema.org structured data from web pages
into typed messages. The vocabulary is syntax-independent, so one generated type
system serves multiple input syntaxes: we target **JSON-LD first, then
Microdata**.

The **vendored schema.org JSON-LD** (`schema/schemaorg-all-https.jsonld`, v30.0)
is the single source of truth. `cmd/genast` builds a **gluon grammar AST in
memory** — each Type a production, each property a reference to its range
production (the cross-entity links) — and projects three artifacts from it:
`proto/schemaorg.{proto,fdset}` (lowered by gluon's compiler) and
`lang/schemaorg.ebnf` (a human-readable view of the grammar). We build the AST
directly rather than round-tripping EBNF text through the markup-terminal
compile machinery proto-svg/proto-html need — the vocabulary is already
structured, so that machinery buys nothing here. **The grammar is central; it is
just built as an AST.** See `docs/decisions/0001-grammar-role.md`.

```
schema/schemaorg-all-https.jsonld        (source of truth, committed)
        │  cmd/genast  (build gluon grammar AST; project ↓)
        ├──────────────┬─────────────────┐
        ▼              ▼                 ▼
proto/schemaorg.proto  proto/….fdset   lang/schemaorg.ebnf   (all COMMITTED)
   (~1841 messages, gluon compiler)     (readable grammar view)
        │
   ┌────┴────┐   runtime front-ends (share the one type system)
   ▼         ▼
 JSON-LD   Microdata          → typed Schema<Type> proto messages
 extractor  extractor (embedded/projection over proto-html; service/ — planned)
```

## Build discipline

**The most important section.**

- **Never build/test/run outside `setup.sh`, `build.sh`, `test.sh`,
  `LET_IT_RIP.sh`.** A bare `go run`/`go build` is fine as a quick compile check
  while editing, but it NEVER counts as validation.
- **Never commit or push without running `./LET_IT_RIP.sh` first**, and never if
  it fails or skips.
- Scripts are idempotent and chained (each runs the previous stage):
  `build.sh → setup.sh`; `test.sh → build.sh`; `LET_IT_RIP.sh → test.sh`.
- **Never hand-edit generated files:** `proto/schemaorg.proto`,
  `proto/schemaorg.fdset`, `lang/schemaorg.ebnf` (and, when added,
  `proto/pb/schemaorg/*.pb.go`). They are regenerated by `tools/genast.sh` (run
  by `build.sh`). To change the grammar/type system, change the **vocabulary**
  (re-vendor a newer schema.org release) or the **generator** (`cmd/genast`),
  then rerun.
- Keep docs in step with the code — no stale documents. Take running notes in
  `docs/claude-worklog/`.

## cmd/genast — the generator

Reads the JSON-LD `@graph` and emits one proto message per schema.org Type:

- **Classes** = `rdfs:Class` nodes; **properties** = `rdf:Property` nodes with
  `schema:domainIncludes` (owning types) and `schema:rangeIncludes` (value
  types); **DataTypes** = `DataType` and everything transitively `subClassOf` it.
- Non-schema.org-native terms (external vocab refs like `bibo:issue`, which carry
  a colon and aren't valid proto identifiers) are filtered out.
- **`Schema<T>`** message = an `id` field (the item's @id / itemid) + one
  **repeated** field per applicable property: the type's own domain **plus every
  ancestor's** (flattened inheritance).
- **Property value:** datatype-only range → `repeated string`; a range admitting
  a nested item → a wrapper message named after the property, holding a `oneof`
  over its class ranges (+ a `text` arm when a DataType is also allowed).
- **Root:** `SchemaOrgDocument { repeated TopLevelItem }`; `TopLevelItem` = a
  `oneof` over every included `Schema<T>`.

**Why references always resolve:** gluon's `compiler.Compile` derives a field's
*name* and its message *type* from the **same** referenced nonterminal name
(snakeCase/pascalCase of one string). The AST references a type by its exact
rule name, so both sides get the identical casing — never hand-roll pascalCase to
"match" gluon; just reuse the same string.

**Scope flags:** `-all` includes every class (918 types → 1841 messages, the
committed default). Otherwise the closure grows from `defaultRoots` through
object-property ranges; `-max N` caps it. Out-of-closure ranges collapse to the
generic `SchemaThing`, so the message graph is always closed.

## Front-ends (planned, `service/`)

Both consume the one generated type system (via `dynamicpb` against
`proto/schemaorg.fdset`) and differ only in parsing. **Parsing goes through the
ecosystem's grammar repos — never hand-rolled byte-level parsing here** (no
`encoding/json`, no `x/net/html`).

- **JSON-LD** (`service/jsonld`, built): `jsonld.Extract(src)` parses raw JSON-LD
  via **accretional/proto-json** (`jsonparse.Parse` → the `json.proto` AST),
  walks that AST, and fills `Schema<T>` messages — a node's `@type` selects the
  message, `@id` fills `id`, each other key is a property (scalar → datatype
  field; nested node → property-wrapper message). This is where real-world
  schema.org data lives.
- **HTML plumbing (pending):** pulling `<script type="application/ld+json">`
  blocks out of a page, and the Microdata DOM-walk (`docs/MICRODATA_SPEC.md`),
  belong to the HTML grammar (**accretional/proto-html**) — a proto-html parser
  library is the prerequisite, not `x/net/html`.

## Testing

- `testing/testdata/synthetic/` — hand-authored HTML + expected-JSON pairs; the
  **ground truth** for the microdata extractor (14 cases, validated against the
  `extruct` reference parser; 2 deliberate spec-vs-extruct differences documented
  there). The extractor must reproduce each `.json` from its `.html`.
- `testing/testdata/live/` — real fetched pages with in-DOM microdata (snapshots;
  may rot). Reported, not gated. Note: microdata is largely extinct on mainstream
  sites — most have moved to JSON-LD (the reason JSON-LD is primary).

## Go module

`github.com/accretional/proto-schemaorg`, go 1.26, with `replace` directives to
sibling checkouts `../gluon` (the AST compiler; use its `v2/` packages) and
`../proto-merge` (descriptor pretty-printer). `setup.sh` clones/updates both.

## History (the grammar, and why it's built as an AST)

Sibling repos (proto-svg, proto-html) make an EBNF *text* grammar the source of
truth and compile it to proto via gluon's `genproto` — markup terminals,
`StripKeywords`, prefix maps, scalarization — machinery that exists so the
grammar can *re-emit real markup* (SVG/HTML bytes). proto-schemaorg keeps the
**grammar** but drops that machinery: schema.org's vocabulary is already a
structured graph and we don't re-emit markup, so `cmd/genast` builds the gluon
grammar AST directly and projects proto + EBNF from it. The grammar is not
absent — it is the AST. Rationale and the context-free-vs-overlay split (where
`itemref` / JSON-LD `@id` cross-references live) are in
`docs/decisions/0001-grammar-role.md`; the full session trail (this repo was
first named proto-microdata) is in `docs/claude-worklog/`.
